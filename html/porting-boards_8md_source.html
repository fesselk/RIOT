    1 Porting boards {#porting-boards}
    2 ================
    3 
    4 At some point you might need to port a new `BOARD` to `RIOT`, either because
    5 that specific development board is not yet supported or because you have a
    6 custom `BOARD` for your project.
    7 
    8 If you want to port a `BOARD` to `RIOT` you have two choices: doing it
    9 inside of `RIOTBASE` or outside. In either case the file structure
    10 is basically the same and moving from one to another is easy.
    11 
    12 This guide details the generic structure you need to add a new `BOARD`
    13 to `RIOT`, the different files as well as their functionality.
    14 
    15 @note We assume here that your `CPU` and `CPU_MODEL` is already supported
    16 in `RIOT` so no peripheral or cpu implementation is needed.
    17 
    18 # General structure {#general-structure}
    19 
    20 Like @ref creating-an-application "applications" or @ref creating-modules
    21 "modules", boards consist on a directory containing source files and
    22 makefiles. Usually a `BOARD` directory has the following structure
    23 
    24 ```
    25  board-foo/
    26  |----dist/
    27  |----scripts
    28  |----board.c
    29  |----doc.txt
    30  |----include/
    31  |----periph_conf.h
    32  |----board.h
    33  |----gpio_params.h
    34  |----Makefile
    35  |----Makefile.dep
    36  |----Makefile.features
    37  |----Makefile.include
    38 ```
    39 
    40 ## Source files {#board-source-files}
    41 
    42 Header files in `board-foo/include` define physical mappings or
    43 configurations. e.g:
    44 
    45 - `periph_conf.h`: defines configurations and mappings for peripherals as well
    46  as clock configurations.
    47 - `board.h`: holds board specific definitions or mappings, for example LEDs,
    48  buttons. It might as well override default drivers parameters (e.g.: assigning
    49  specific pin connections to a LCD screen, radio, etc.). Some boards might also
    50  define optimized `XTIMER_%` values (e.g. @ref XTIMER_BACKOFF).
    51 - `gpio_params.h`: if the board supports @ref drivers_saul "SAUL" then its
    52  @ref saul_gpio_params_t is defined here.
    53 - other: other specific headers needed by one `BOARD`
    54 
    55 @note Header files do not need to be defined in `include/`, but if defined
    56 somewhere else then they must be added to the include path. In
    57 `Makefile.include`: `INCLUDES += -I<some>/<directory>/<path>`
    58 
    59 Board initialization functions are defined in `board.c`. This file must at
    60 least define a `board_init()` function that is called at startup. This
    61 function initializes the `CPU` by calling`cpu_init()` among others.
    62 
    63 ```c
    64 void board_init(void)
    65 {
    66  /* initialize the CPU core */
    67  cpu_init();
    68 
    69  /* initialize GPIO or others... */
    70  ...
    71 }
    72 ```

    74 ## Makefiles
    75 
    76 ### Makefile {#Makefile}
    77 
    78 A board's Makefile just needs to include `Makefile.base` in the RIOT
    79 repository and define the `MODULE` as `board` (see @ref creating-modules
    80 "modules" for more details)
    81 
    82 ```mk
    83 MODULE = board
    84 
    85 include $(RIOTBASE)/Makefile.base
    86 ```
    87 
    88 ### Makefile.dep {#makefile-dep}
    89 
    90 Dependencies on other `MODULES` or `FEATURES` can be defined here. This might
    91 specify `MODULES` or dependencies that need to be pulled under specific
    92 configurations. e.g.: if your board has a sx1276 lora chip:
    93 
    94 ```mk
    95 ifneq (,$(filter netdev_default,$(USEMODULE)))
    96  USEMODULE += sx1276
    97 endif
    98 ```
    99 
    100 @note `Makefile.dep` is processed only once so you have to take care of adding
    101 the dependency block for your board *before* its dependencies pull in their own
    102 dependencies.
    103 
    104 ### Makefile.features {#makefile-features}
    105 
    106 This file defines all the features provided by the BOARD. These features
    107 might also need to be supported by the `CPU`. Here, define the `CPU` and
    108 `CPU_MODEL` (see @ref build-system-basics "build system basics" for more details
    109  on these variables).
    110 
    111 e.g.:
    112 
    113 ```mk
    114 CPU = foo
    115 CPU_MODEL = foobar
    116 
    117 # Put defined MCU peripherals here (in alphabetical order)
    118 FEATURES_PROVIDED += periph_i2c
    119 FEATURES_PROVIDED += periph_spi
    120 FEATURES_PROVIDED += periph_uart
    121 ```
    122 
    123 ### Makefile.include {#makefile-include}
    124 
    125 This file contains BSP or toolchain configurations for the `BOARD`. It
    126 should at least define the configuration needed for flashing (i.e. a
    127 programmer) as well as the serial configuration (if one is available).
    128 
    129 e.g.:
    130 
    131 ```mk
    132 # Define the default port depending on the host OS
    133 PORT_LINUX ?= /dev/ttyUSB0
    134 PORT_DARWIN ?= $(firstword $(sort $(wildcard /dev/tty.usbserial*)))
    135 
    136 # setup serial terminal
    137 include $(RIOTMAKE)/tools/serial.inc.mk
    138 
    139 # this board uses openocd
    140 include $(RIOTMAKE)/tools/openocd.inc.mk
    141 ```
    142 
    143 ## doc.txt {#board-doc}
    144 
    145 Although not explicitly needed, if upstreamed and as a general good
    146 practice, this file holds all `BOARD` documentation. This can include
    147 datasheet reference, documentation on how to flash, etc.
    148 
    149 The documentation must be under the proper doxygen group, you can compile the
    150 documentation by calling `make doc` and then open the generated html file on
    151 any browser.
    152 
    153 @code
    154 /**
    155 @defgroup boards_foo FooBoard
    156 @ingroup boards
    157 @brief Support for the foo board
    158 @author FooName BarName <foo.bar@baz.com>
    159 
    160 ### User Interface
    161 
    162  ....
    163 
    164 ### Using UART
    165 
    166  ...
    167 
    168 ### Flashing the device
    169 
    170  ...
    171 
    172 */
    173 @endcode

    175 # Helper tools
    176 
    177 To help you start porting a board, the RIOT build system provides the
    178 `generate-board` make target. It is a wrapper around the
    179 [riotgen](https://pypi.org/project/riotgen/) command line tool that is helpful
    180 when starting to port a board: all required files are generated with
    181 copyright headers, doxygen groups, etc, so you can concentrate on the port.
    182 The board source files are created in the `boards/<board name>` directory.
    183 
    184 **Usage:**
    185 
    186 From the RIOT base directory, run:
    187 ```
    188 make generate-board
    189 ```
    190 Then answer a few questions about the driver:
    191 - Board name: Enter a name for your board. It will be used as the name
    192  of the board directory under `boards`.
    193 - Board displayed name: Enter the name of the board, as displayed in the
    194  Doxygen documentation.
    195 - CPU name: Enter the name of the CPU embedded on the board.
    196 - CPU model name: Enter the precise model name of the CPU.
    197 - Features provided: CPU features provided (and configured) for this board.
    198 
    199 Other global information (author name, email, organization) should be retrieved
    200 automatically from your git configuration.

    202 # Using Common code {#common-board-code}
    203 
    204 To avoid code duplication, common code across boards has been grouped in
    205 `boards/common`. e.g. `BOARD`s based on the same cpu (`boards/common/nrf52`) or
    206 `BOARD`s having the same layout `boards/common/nucleo64`.
    207 
    208 In the case of source files this means some functions like `board_init` can be
    209 already defined in the common code. Unless having specific configurations or
    210 initialization you might not need a `board.c` or `board.h`. Another common use
    211 case is common peripheral configurations:
    212 
    213 @code
    214 -\#include "cfg_timer_tim5.h"
    215 +/**
    216 + * @name Timer configuration
    217 + * @{
    218 + */
    219 +static const timer_conf_t timer_config[] = {
    220 + {
    221 + .dev = TIM5,
    222 + .max = 0xffffffff,
    223 + .rcc_mask = RCC_APB1ENR_TIM5EN,
    224 + .bus = APB1,
    225 + .irqn = TIM5_IRQn
    226 + }
    227 +};
    228 +
    229 +#define TIMER_0_ISR isr_tim5
    230 +
    231 +#define TIMER_NUMOF ARRAY_SIZE(timer_config)
    232 +/** @} */
    233 @endcode
    234 
    235 If you want to use common makefiles, include them at the end of the specific
    236 `Makefile`, e.g. for a `Makefile.features`:
    237 
    238 ```mk
    239 CPU = foo
    240 CPU_MODEL = foobar
    241 
    242 # Put defined MCU peripherals here (in alphabetical order)
    243 FEATURES_PROVIDED += periph_i2c
    244 FEATURES_PROVIDED += periph_spi
    245 FEATURES_PROVIDED += periph_uart
    246 
    247 include $(RIOTBOARD)/common/foo_common/Makefile.features
    248 ```

    250 # Boards outside of RIOTBASE {#boards-outside-of-riotbase}
    251 
    252 All `BOARD`s in RIOT reside in `RIOTBOARD` (`RIOTBOARD` being a make variable
    253 set to `$(RIOTBOARD)/boards`).
    254 
    255 If one wants to use a `BOARD` outside of `RIOTBOARD`, the way to go is setting
    256 the `EXTERNAL_BOARD_DIRS` variable to the path to the directory containing your
    257 external boards, e.g.: `EXTERNAL_BOARD_DIRS=/home/external-boards/` (this would
    258 commonly be done in your application `Makefile` or your environment). You can
    259 specify multiple directories separated by spaces.
    260 
    261 ```
    262 /home/
    263 |----RIOT/
    264  |---- ...
    265 |----external-boards/
    266  |----board-foo/
    267  |----dist/
    268  |----scripts
    269  |----board.c
    270  |----doc.txt
    271  |----include/
    272  |----periph_conf.h
    273  |----board.h
    274  |----gpio_params.h
    275  |----Makefile
    276  |----Makefile.dep
    277  |----Makefile.features
    278  |----Makefile.include
    279 ```
    280 
    281 If the external `BOARD` is very similar to a `BOARD` already present in
    282 `RIOTBOARD`, the external `BOARD` (`board-foo`) can inherit from that
    283 parent `BOARD` (e.g: `foo-parent`).
    284 
    285 In this case some special considerations must be taken with the makefiles:
    286 
    287 - `Makefile`
    288  - `MODULE` cannot be `board`: `foo-parent` will already define
    289  `MODULE = board`, so use any other name, lets say `MODULE = board-foo`.
    290  - Include the location of the parent `BOARD` to inherit from (if there is
    291  one):
    292 
    293 ```mk
    294  DIRS += $(RIOTBOARD)/foo-parent
    295 ```
    296 
    297 - `Makefile.include`
    298  - duplicate the include done by `$(RIOTBASE)/Makefile.include` to also
    299  include the parent board header. e.g: if inheriting from `foo-parent`
    300  ``INCLUDES += $(addprefix -I,$(wildcard $(RIOTBOARD)/foo-parent/include))`
    301 
    302 - `Makefile.dep`: `board` is added by default to `USEMODULE` but since
    303  `board-foo` is used for this `BOARD`, it must be explicitly included by adding
    304  `USEMODULE += board-foo`.
    305 
    306 - Then simply include in each `Makefile.*` the corresponding parent `BOARD`
    307  `Makefile.*`, just as it is done for common `BOARD` code (as explained in
    308  @ref common-board-code). e.g:
    309  `include $(RIOTBOARD)/foo-parent/Makefile.*include*`
    310 
    311 An example can be found in
    312 [`tests/external_board_native`](https://github.com/RIOT-OS/RIOT/tree/master/tests/external_board_native)

    314 # Tools {#boards-tools}
    315 
    316 Some scripts and tools available to ease `BOARD` porting and testing:
    317 
    318  - Run `dist/tools/insufficient_memory/add_insufficient_memory_board.sh <board>`
    319  if your board has little memory. This updates the `Makefile.ci` lists to
    320  exclude the `BOARD` from automated compile-tests of applications that do
    321  not fit on the `BOARD`s `CPU`.
    322 
    323  - Run `dist/tools/compile_and_test_for_board/compile_and_test_for_board.py . <board> --with-test-only`
    324  to run all automated tests on the new board.
