    6 RIOT in a nutshell {#riot-in-a-nutshell}
    7 ==================
    8 RIOT is an open-source microkernel-based operating system, designed to match
    9 the requirements of Internet of Things (IoT) devices and other embedded
    10 devices. These requirements include a very low memory footprint (on the order
    11 of a few kilobytes), high energy efficiency, real-time capabilities, support
    12 for a wide range of low-power hardware, communication stacks for wireless and
    13 communication stacks for wired networks.
    14 
    15 RIOT provides a microkernel, multiple network stacks, and utilities which
    16 include cryptographic libraries, data structures (bloom filters, hash tables,
    17 priority queues), a shell and more. RIOT supports a wide range of
    18 microcontroller architectures, radio drivers, sensors, and configurations for
    19 entire platforms, e.g. Atmel SAM R21 Xplained Pro, Zolertia Z1, STM32 Discovery
    20 Boards etc. (see the list of
    21 [supported hardware](https://github.com/RIOT-OS/RIOT/wiki/RIOT-Platforms).
    22 Across all supported hardware (32-bit, 16-bit, and 8-bit platforms). RIOT
    23 provides a consistent API and enables ANSI C and C++ application programming,
    24 with multithreading, IPC, system timers, mutexes etc.
    25 
    26 A good high-level overview can be found in the article
    27 [RIOT: An Open Source Operating System for Low-End Embedded Devices in
    28 the IoT](https://riot-os.org/docs/riot-ieeeiotjournal-2018.pdf)
    29 (IEEE Internet of Things Journal, December 2018).

    31 Contribute to RIOT {#contribute-to-riot}
    32 ==================
    33 RIOT is developed by an open community that anyone is welcome to join:
    34  - Download and contribute your code on
    35  [GitHub](https://github.com/RIOT-OS/RIOT). You can read about how to
    36  contribute [in our GitHub
    37  Wiki](https://github.com/RIOT-OS/RIOT/wiki/Contributing-to-RIOT).
    38  - [Subscribe](http://lists.riot-os.org/mailman/listinfo/users) to
    39  users@riot-os.org to ask for help using RIOT or writing an application for
    40  RIOT (or to just stay in the loop). An archive of this list [is available
    41  here](https://lists.riot-os.org/pipermail/users/).
    42  - [Subscribe](http://lists.riot-os.org/mailman/listinfo/devel) to
    43  devel@riot-os.org to follow and discuss kernel and network stack
    44  development, or hardware support. An archive of this list [is available
    45  here](https://lists.riot-os.org/pipermail/devel/).
    46  - Follow us on [Twitter](https://twitter.com/RIOT_OS) for news from the RIOT
    47  community.
    48  - Regarding critical vulnerabilities we would appreciate if you give us a
    49  90-days head-start by reporting to security@riot-os.org, before making your
    50  information publicly available
    51  - Contact us on IRC for live support and discussions:
    52  [irc.freenode.org \#riot-os](irc://irc.freenode.org/riot-os)

    55 The quickest start {#the-quickest-start}
    56 ==================
    57 You can run RIOT on most IoT devices, on open-access testbed hardware (e.g.
    58 IoT-lab), and also directly as a process on your Linux/FreeBSD/OSX machine (we
    59 call this the `native` port). Try it right now in your terminal window:
    60 
    61 ~~~~~~~{.sh}
    62 git clone git://github.com/RIOT-OS/RIOT.git # assumption: git is pre-installed
    63 cd RIOT
    64 git checkout <LATEST_RELEASE>
    65 sudo ./dist/tools/tapsetup/tapsetup # create virtual Ethernet
    66  # interfaces to connect multiple
    67  # RIOT instances
    68 cd examples/default/
    69 make all
    70 make term
    71 ~~~~~~~
    72 
    73 ... and you are in the RIOT shell!
    74 Type `help` to discover available commands. For further information see the
    75 [README of the `default` example](https://github.com/RIOT-OS/RIOT/tree/master/examples/default).
    76 
    77 To use RIOT directly on your embedded platform, and for more hands-on details
    78 with RIOT, see @ref getting-started.

    83 Structure {#structure}
    84 =========
    85 
    86 This section walks you through RIOT's structure. Once you understand this
    87 structure, you will easily find your way around in RIOT's code base.
    88 
    89 ![Overview](riot-structure.svg)
    90 
    91 RIOT's code base is structured into five groups.
    92 
    93  - The kernel (`core`)
    94  - Platform specific code (`cpu`; `boards`)
    95  - Device drivers (`drivers`)
    96  - Libraries and network code (`sys`; `pkg`)
    97  - Applications for demonstrating features and for testing (`examples`;
    98  `tests`)
    99 
    100 In addition RIOT includes a collection of scripts for various tasks (`dist`) as
    101 well as a predefined environment for generating this documentation (`doc`).
    102 
    103 The structural groups are projected onto the directory structure of RIOT, where
    104 each of these groups resides in one or two directories in the main RIOT
    105 directory.
    106 
    107 The following list gives a more detailed description of each of RIOT's
    108 top-level directories:
    109 
    110 core
    111 ----
    112 This directory contains the actual kernel. The kernel consists of the scheduler,
    113 inter-process-communication (messaging), threading, and thread
    114 synchronization, as well as supporting data-structures and type definitions.
    115 
    116 See @ref core for further information and API documentations.
    117 
    118 boards
    119 ------
    120 The platform dependent code is split into two logic elements: CPUs and boards,
    121 while maintaining a strict 1-to-n relationship, a board has exactly one CPU,
    122 while a CPU can be part of n boards. The CPU part contains all generic, CPU
    123 specific code (see below).
    124 
    125 The board part contains the specific configuration for the CPU it contains.
    126 This configuration mainly includes the peripheral configuration and
    127 pin-mapping, the configuration of on-board devices, and the CPU's clock
    128 configuration.
    129 
    130 On top of the source and header files needed for each board, this directory
    131 additionally may include some script and configuration files needed for
    132 interfacing with the board. These are typically custom flash/debug scripts or
    133 e.g. OpenOCD configuration files. For most boards, these files are located in a
    134 `dist` sub-directory of the board.
    135 
    136 See here @ref boards for further information.
    137 
    138 cpu
    139 ---
    140 For each supported CPU this directory contains a sub-directory with the name of
    141 the CPU. These directories then contain all CPU specific configurations, such
    142 as implementations of power management (LPM), interrupt handling and vectors,
    143 startup code, clock initialization code and thread handling (e.g. context
    144 switching) code. For most CPUs you will also find the linker scripts in the
    145 `ldscripts` sub-directory.
    146 
    147 In the `periph` sub-directory of each CPU you can find the implementations of
    148 the CPU's peripheral drivers like SPI, UART, GPIO, etc. See @ref drivers_periph
    149 for their API documentation.
    150 
    151 Many CPUs share a certain amount of their code (e.g. all ARM Cortex-M based
    152 CPUs share the same code for task switching and interrupt handling). This
    153 shared code is put in its own directories, following a `xxxxx_common` naming
    154 scheme. Examples for this is code shared across architectures (e.g.
    155 `cortexm_common`, `msp430_comon`) or code shared among vendors (e.g.
    156 `stm32_common`).
    157 
    158 See @ref cpu for more detailed information.
    159 
    160 drivers
    161 -------
    162 This directory contains the drivers for external devices such as network
    163 interfaces, sensors and actuators. Each device driver is put into its own
    164 sub-directory with the name of that device.
    165 
    166 All of RIOT's device drivers are based on the peripheral driver API (e.g. SPI,
    167 GPIO, etc.) and other RIOT modules like the `xtimer`. This way the drivers are
    168 completely platform agnostic and they don't have any dependencies into the CPU
    169 and board code.
    170 
    171 See @ref drivers for more details.
    172 
    173 sys
    174 ---
    175 RIOT follows the micro-kernel design paradigm where everything is supposed to
    176 be a module. All of these modules that are not part of the hardware abstraction
    177 nor device drivers can be found in this directory. The libraries include data
    178 structures (e.g. bloom, color), crypto libraries (e.g. hashes, AES) ,
    179 high-level APIs (e.g. Posix implementations), memory management (e.g. malloc),
    180 the RIOT shell and many more.
    181 
    182 See @ref sys for a complete list of available libraries
    183 
    184 sys/net
    185 -------
    186 The `sys/net` sub-directory needs to be explicitly mentioned, as this is where
    187 all the networking code in RIOT resides. Here you can find the network stack
    188 implementations (e.g. the @ref net_gnrc "GNRC" stack) as well as network stack agnostic code as
    189 header definitions or network types.
    190 
    191 See @ref net for more details on networking code.
    192 
    193 pkg
    194 ---
    195 RIOT comes with support for a number of external libraries (e.g.
    196 [ccn-lite](https://github.com/cn-uofbasel/ccn-lite),
    197 [microcoap](https://github.com/1248/microcoap)). The way they are included is
    198 that RIOT ships with a custom Makefile for each supported library that
    199 downloads the library and optionally applies a number of patches to make it
    200 work with RIOT. These Makefiles and patches can be found in the `pkg`
    201 directory.
    202 
    203 See @ref pkg for a detailed description on how this works.
    204 
    205 examples
    206 --------
    207 Here you find a number of example applications that demonstrate certain
    208 features of RIOT. The default example found in this directory is a good
    209 starting point for anyone who is new to RIOT.
    210 
    211 For more information best browse that directory and have a look at the
    212 `README.md` files that ship with each example.
    213 
    214 To create your own application - here or anywhere else - see @ref creating-an-application
    215 
    216 tests
    217 -----
    218 Many features/modules in RIOT come with their own test application, which are
    219 located in this directory. In contrary to the examples these tests are mostly
    220 focusing on a single aspect than on a set of features. Despite for testing, you
    221 might consider these tests also for insights on understanding RIOT.
    222 
    223 dist & doc
    224 ----------
    225 All the tooling around RIOT can be found in these two folders.
    226 
    227 `doc` contains the doxygen configuration and also contains the compiled doxygen
    228 output after running `make doc`.
    229 
    230 Lastly, the `dist` directory contains tools to help you with RIOT. These
    231 include
    232 the serial terminal application `pyterm`, generic scripts for flashing,
    233 debugging, resetting (e.g. support for [OpenOCD](http://openocd.org/),
    234 [Jlink](https://www.segger.com/jlink_base.html)), as well as code enabling easy
    235 integration to open testbeds such as the [IoT-LAB](https://www.iot-lab.info/).
    236 Furthermore you can find here scripts to do all kind of code and style checks.
