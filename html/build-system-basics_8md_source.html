    5 ## FEATURES {#features}
    6 
    7 ### What is a FEATURE?
    8 
    9 A `FEATURE` is a mean of specifying valid/invalid dependencies and configurations.
    10 
    11 Whenever a `FEATURE` is used there should be at some level a hardware requirement,
    12 whether this is a _radio_, a _bus_ of a specific core architecture.
    13 
    14 This is not a hard line, in some cases the line can be harder to establish than
    15 others. There are complicated cases like `netif` since a network interface could
    16 be fully implemented in software as a loop-back.
    17 
    18 It's also important to note that a `FEATURE` does not mean there is a `MODULE`
    19 with the same name. There could be many implementations for the same `FEATURE`.
    20 The fact that many `FEATURES` translate directly into a `MODULE` is only by
    21 convenience.
    22 
    23 e.g.
    24 
    25  # all periph features correspond to a periph submodule
    26  USEMODULE += $(filter periph_%,$(FEATURES_USED))
    27 
    28 ### Providing a FEATURE
    29 
    30 For a `FEATURE` to be provided by a `board` it must meet 2 criteria, and for
    31 `periph_%` and other _hw_ (hardware) related `FEATURES` it must follow a 3rd criteria.
    32 
    33 - Needs the "hardware" or BSP support (toolchain, build system, flasher, etc.)
    34  - e.g.: `stm32l152re` has an SPI peripheral
    35  `riotboot` needs to be able to link and flash at an offset
    36 - Needs support in RIOT, an implementation of an api to interact with the _hw_
    37  - e.g.: `cpu/stm32_common/periph/spi.c` is implemented for `stm32l1`
    38  `riotboot` needs an implementation of `cpu_jump_to_image`
    39 - Wiring between the _cpu/soc_(system on a chip) a _bus_ and other _cpu_/_hw_ components.
    40  - e.g.: `nucleo-l152re/include/periph_conf.h` specified wiring between `PORT_Ax`
    41  and `SPI1`
    42 
    43 ### All the FEATURES_%
    44 
    45 - `FEATURES_PROVIDED` are available hardware (including BSP) features
    46  (e.g.:`periph_hwrng`, `periph_uart`) or characteristics (e.g:`arch_32bits`) of
    47  a board.
    48 
    49 - `FEATURES_CONFLICT` are a series of `FEATURES` that can't be used at the same
    50  time for a particular `BOARD`.
    51 
    52 - `FEATURES_REQUIRED` are `FEATURES` that are needed by a `MODULE` or `APPLICATION`
    53  to work.
    54 
    55 - `FEATURES_OPTIONAL` are "nice to have" `FEATURES`, not needed but useful. If
    56  available they are always included.
    57 
    58 - `FEATURES_REQUIRED_ANY` are `FEATURES` of which (at least) one of
    59  is needed by a `MODULE` or `APPLICATION`. Alternatives are separated by
    60  a pipe (`|`) in order of preference, e.g.:
    61  `FEATURES_REQUIRED_ANY += arch_avr8|arch_native` if both are provide then
    62  `arch_avr8` will be used.
    63 
    64 - `FEATURES_BLACKLIST` are `FEATURES` that can't be used by a `MODULE` or `APPLICATION`.
    65  They are usually used for _hw_ characteristics like `arch_` to easily resolve
    66  unsupported configurations for a group.
    67 
    68 - `FEATURES_USED` are the final list of `FEATURES` that will be used by an `APPLICATION`
    69 
    70 ### Where to define FEATURES_%
    71 
    72 - `FEATURES_PROVIDED`, `FEATURES_CONFLICT` and `FEATURES_CONFLICT_MSG ` are
    73  defined in `Makefile.features`
    74 
    75 - `FEATURES_REQUIRED`, `FEATURES_OPTIONAL`, `FEATURES_REQUIRED_ANY`,
    76  and `FEATURES_BLACKLIST` are defined by the application `Makefile`
    77  (`examples/%/Makefile`, `tests/%/Makefile`, etc.) or in `Makefile.dep`

    79 ## CPU/CPU_MODEL {#cpu}
    80 
    81 
    82 `CPU` and `CPU_MODEL` refer to the _soc_ or _mcu_ (microcontroller)
    83 present in a `BOARD`. The variables `CPU`, `CPU_FAM`, etc. are just arbitrary groupings
    84 to avoid code duplication. How this grouping is done depends on every implementation
    85 and the way each manufacturer groups there products.
    86 
    87 These variables allows declaring the `FEATURES` that the _mcu/soc_ provides as well
    88 as resolving dependencies.
    89 
    90 `FEATURES` provided by a `CPU/CPU_MODEL` should not depend on the wiring of a
    91 specific `BOARD` but be intrinsic to the _soc/mcu_.
    92 
    93 A `CPU/CPU_MODEL` might support `FEATURES` that will depend on the `BOARD` wiring,
    94 e.g.: bus (`uart`, `spi`) mappings. In this cases the `FEATURE` should be provided
    95 by the `BOARD.`
    96 
    97 ## BOARD {#board}
    98 
    99 In RIOTs build-system, a `BOARD` is a grouping of:
    100 
    101 - _soc/mcu_ (`CPU/CPU_MODEL`)
    102  - e.g.: `b-l072z-lrwan1` `stm32l072cz`
    103 - _sensor/actuators_ (buttons and leds included) (`drivers`)
    104  - e.g.: `b-l072z-lrwan1` leds and buttons
    105 - _radios_, _ethernet_, etc. devices (`drivers`)
    106  - e.g.: `b-l072z-lrwan1` `sx1276`
    107 - _programming/debugging_ tools
    108  - e.g.: `b-l072z-lrwan1` `stlink`
    109 - configuration mapping cpu support capabilities to availability
    110  - e.g.: `b-l072z-lrwan1` `periph_conf.h`, `gpio_params`
    111 
    112 A `board` can have all the required `FEATURES` to interact with a radio or
    113 sensor/actuator, but it doesn't necessarily provide that `FEATURE`.
    114 
    115 e.g.:
    116  - `samr21-xpro` provides a `at86rf233` radio as well as the necessary
    117  `periph_*` features.
    118  - `nucleo-*` provide all `periph_*` features to use `sx1272`, and
    119  even a default configuration for the `SX1272MB2xA` shield, but not
    120  doesn't provide the radio.
    121 
    122 If a `board` in `$(RIOTBASE)/boards` is connected to a radio shield, sensors,
    123 actuators, etc. then it is a different `board` than the one provided by default.
    124 Whenever you need to have a device mapping (in linux-arm, it would require
    125 a different device tree), then it is a different board and would need a
    126 different `board/periph_conf`.
    127 
    128 A `nucleo-*` with a `SX1272MB2xA` is a different board in RIOT sense.
    129 
    130 _note_: if `devicetree` is implemented this concept will change.

    133 # Variables declaration guidelines {#variable-declaration-guidelines}
    134 
    135 This page contains basic guidelines about `make` variable declaration, it
    136 summarizes some of the pros and cons as well as specifies good and bad patterns
    137 in our build system. You might want to refer to `gnu make` documentation
    138 regarding these subjects.
    139 
    140 ## Avoid unnecessary export
    141 
    142 ```
    143 export OUTPUT = $(shell some-command)
    144 ```
    145 
    146 Exporting a variable means it will be evaluated on every `target` call, which
    147 slows down the build system. Always avoid exporting a variable if unneeded.
    148 
    149 If an export is actually needed by a `sub-make` then export the variable only for
    150 the needed targets using `target-export-variables` (more in
    151 `makefiles/utils/variables.mk`).
    152 
    153 Exported variables ("global variable") are hard to remove, specially when badly
    154 documented. If no one knows why it's there and no one knows where it can be used
    155 then no one knows if it's safe to remove since it's present for every target.
    156 This is why global variables need clear documentation.
    157 
    158 [gnumake doc](https://www.gnu.org/software/make/manual/html_node/Variables_002fRecursion.html)
    159 
    160 ## Use memoized for variables referencing a function or command
    161 
    162 ### recursively expanded variable:
    163 
    164 ~~~~~~~~~~~~~~~~
    165 OUTPUT = $(shell some-command $(ANOTHER_VARIABLE))
    166 ~~~~~~~~~~~~~~~~
    167 
    168 - When using `=` the value of the variable is only declared, but not set,
    169  therefore the variable will only be evaluated when expanded (used) somewhere
    170  in the makefile. If `$(OUTPUT)` is never expanded, `some-command`
    171  is never executed and `ANOTHER_VARIABLE` not expanded.
    172 
    173 - All variables or functions referenced by the declared variable will will be
    174  evaluated every time the variable is expanded.
    175  In the example `some-command` is executed every time `OUTPUT` is expanded, same for
    176  `ANOTHER_VARIABLE`. If `some-command` is slow this introduced unneeded overhead.
    177 
    178 - If the variable expansion doesn't involve evaluating a function the overhead
    179  is none.
    180 
    181 ### simply expanded variable:
    182 
    183 ~~~~~~~~~~~~~~~~
    184 OUTPUT := $(shell some-command $(ANOTHER_VARIABLE))
    185 ~~~~~~~~~~~~~~~~
    186 
    187 - When using `:=` the value is only expanded once, expanding any reference to
    188  other variables or functions. If `OUTPUT` is always used at least once and
    189  evaluates a costly function (`some command`) then use `:=`.
    190 
    191 - When using `:=` the variable will be evaluated even if not needed, which
    192  introduces unnecessary delay, in particular `some command` or functions
    193  evaluated by `ANOTHER_VARIABLE` are slow.
    194  It can also cause a failure in a worst-case scenario (think what happens if a
    195  tool is defined with `:=` but you don't have the tool and you don't need it either).
    196 
    197 - The values of variables declared with `:=` depend on the order of definition.
    198 
    199 ### memoized:
    200 
    201 ~~~~~~~~~~~~~~~~
    202 OUTPUT = $(call memoized,OUTPUT,$(shell some-command))
    203 ~~~~~~~~~~~~~~~~
    204 
    205 - `memoized` is a RIOT defined function that combines characteristics from
    206  both `=` and `:=`.
    207  The variable expansion will be deferred until its first usage, but further
    208  usage will consider it as a simply expanded variable, so it will use the already
    209  evaluated value. In the example `some-command` would be executed once or not
    210  at all (more in `makefiles/utils/variables.mk`).
    211 
    212 [gnumake doc](https://www.gnu.org/software/make/manual/html_node/Flavors.html)
