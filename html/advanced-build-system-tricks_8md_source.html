    6 Introduction {#introduction}
    7 ============
    8 
    9 This page describes some build systems tricks that can help developers but are
    10 not part of the standard workflow.
    11 
    12 They are low level commands that should not be taken as part of a stable API
    13 but better have a documentation than only having a description in the build
    14 system code.
    15 
    16 
    17 Customize the build system {#customize-build-system}
    18 ==========================
    19 
    20 + `RIOT_MAKEFILES_GLOBAL_PRE`: files parsed before the body of
    21  `$RIOTBASE/Makefile.include`
    22 + `RIOT_MAKEFILES_GLOBAL_POST`: files parsed after the body of
    23  `$RIOTBASE/Makefile.include`
    24 
    25 The variables are a list of files that will be included by
    26 `$RIOTBASE/Makefile.include`.
    27 They will be handled as relative to the application directory if the path is
    28 relative.
    29 
    30 
    31 Usage
    32 -----
    33 
    34 You can configure your own files that will be parsed by the build system main
    35 `Makefile.include` file before or after its main body, examples usages can be:
    36 
    37 * Globally overwrite a variable, like `TERMPROG`
    38 * Specify a hard written `PORT` / `DEBUG_ADAPTER_ID` for some BOARD values
    39 * Define your custom targets
    40 * Override default targets
    41

    43 Handling multiple boards with udev-rules {#multiple-boards-udev}
    44 ========================================
    45 
    46 When developing and working with multiple boards the default `PORT` configuration
    47 for a particular board might not apply anymore so `PORT` will need to be specified
    48 whenever calling `make term/test`. This can also happen if multiple `DEBUGGERS/PROGRAMMERS`
    49 are present so `DEBUG_ADAPTER_ID` will also need to be passed. Keeping track of
    50 this will become annoying.
    51 
    52 One way of handling this is to use `udev` rules to define `SYMLINKS` between the
    53 boards serial port (`riot/tty-<board-name>`) and the actual serial port
    54 (dev/ttyACM* or other). With this we can query the rest of the boards serial
    55 `dev` information (`DEBUG_ADAPTER_ID`, `PORT`, etc.) to always flash and open a
    56 terminal on the correct port.
    57 
    58 Procedure:
    59 
    60 - use `udevadm info /dev/ttyACM0` to query the udev database for information on
    61  device on port `/dev/ttyACM0`.
    62 
    63  or use `udevadm info --attribute-walk --name /dev/ttyACM0` for more detailed
    64  output when the first level of information isn't enough
    65 
    66 - create a udev rule with information of the device and one parent to create a
    67  matching rule in `/etc/udev/rules.d/70-riotboards.rules`.
    68 
    69 ~~~~~~~~~~~~~~~~~~~
    70  # samr21-xpro
    71  SUBSYSTEM=="tty", SUBSYSTEMS=="usb", ATTRS{idVendor}=="03eb", \
    72  ATTRS{idProduct}=="2111", ATTRS{manufacturer}=="Atmel Corp.", \
    73  ATTRS{serial}=="ATML2127031800004957", SYMLINK+="riot/tty-samr21-xpro"
    74 ~~~~~~~~~~~~~~~~~~~
    75 
    76 - reload rules: `udevadm control --reload-rules`
    77 
    78 - Boards `PORT` are symlinked to /dev/riot/tty-`board-name`.
    79 
    80 - Create a `makefile.pre` that will query the real `PORT` and the `DEBUG_ADAPTER_ID`
    81  from the `SYMLINK` info
    82 
    83 ~~~~~~~~~~~~~~~~~~~
    84  PORT = /dev/riot/tty-$(BOARD)
    85  DEBUG_ADAPTER_ID = $(\
    86  shell udevadm info -q property $(PORT) |\
    87  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    88 ~~~~~~~~~~~~~~~~~~~
    89 
    90 - You can now add `makefile.pre` to `RIOT_MAKEFILES_GLOBAL_PRE` as an environment
    91  variable or on each `make` call:
    92 
    93 ~~~~~~~~~~~~~~~~~~~
    94  $ RIOT_MAKEFILES_GLOBAL_PRE=/path/to/makefile.pre make -C examples/hello-world flash term
    95 ~~~~~~~~~~~~~~~~~~~
    96 
    97 _note_: if set as an environment variable it would be a good idea to add a variable
    98 to enable/disable it, e.g:
    99 
    100 ~~~~~~~~~~~~~~~~~~~
    101 ifeq (1,$(ENABLE_LOCAL_BOARDS))
    102  PORT = /dev/riot/tty-$(BOARD)
    103  DEBUG_ADAPTER_ID = $(\
    104  shell udevadm info -q property $(PORT) |\
    105  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    106 endif
    107 ~~~~~~~~~~~~~~~~~~~
    108 
    109 Handling multiple versions of the same BOARD
    110 -------------------------------------------
    111 
    112 The above procedure works fine when handling different boards, but not
    113 multiple times the same board, e.g: multiple `samr21-xpro`.
    114 
    115 An option for this would be to add an identifier of that board to the mapped
    116 `riot/tty-*`, there are multiple ways of handling this but in the end it means
    117 having a way to identify every copy.
    118 
    119 Another way would be to map the `DEBUG_ADAPTER_ID` in the name:
    120 
    121 ~~~~~~~~~~~~~~~~~~~
    122 SYMLINK+="riot/node-$attr{serial}
    123 ~~~~~~~~~~~~~~~~~~~
    124 
    125 But it will require to know in advance the serial number of each board you want
    126 to use. Another option would be to add some kind of numbering and defining
    127 multiple symlinks for each board. e.g. for `samr21-xpro` number `n`:
    128 
    129 ~~~~~~~~~~~~~~~~~~~
    130  # samr21-xpro
    131  SUBSYSTEM=="tty", SUBSYSTEMS=="usb", ATTRS{idVendor}=="03eb", \
    132  ATTRS{idProduct}=="2111", ATTRS{manufacturer}=="Atmel Corp.", \
    133  ATTRS{serial}=="ATML2127031800004957", SYMLINK+="riot/tty-samr21-xpro", \
    134  SYMLINK+="riot/tty-samr21-xpro-n"
    135 ~~~~~~~~~~~~~~~~~~~
    136 
    137 Then, when flashing, the number can be specified and the parsing adapted:
    138 
    139 ~~~~~~~~~~~~~~~~~~~
    140  ifneq(,$(BOARD_NUM))
    141  PORT = /dev/riot/tty-$(BOARD)-$(BOARD_NUM)
    142  else
    143  PORT = /dev/riot/tty-$(BOARD)
    144  endif
    145  DEBUG_ADAPTER_ID = $(\
    146  shell udevadm info -q property $(PORT) |\
    147  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    148 ~~~~~~~~~~~~~~~~~~~
    149 
    150 ~~~~~~~~~~~~~~~~~~~
    151  BOARD=samr21-xpro BOARD_NUM=n make flash term
    152 ~~~~~~~~~~~~~~~~~~~
    153 
    154 In the end, this would be the same as using the serial, but a simple number might
    155 be easier to handle.
    156 
    157 Notes
    158 -----
    159 Udev only parses SUBSYSTEM and one parent. For others, we will rely on ENV
    160 variables defined by 60-serial.rules
    161 
    162 So the current filename should be higher than 60-serial.rules
    163 
    164 If for some reason re-writing the serial is needed there is a windows tool:
    165  https://remoteqth.com/wiki/index.php?page=How+to+set+usb+device+SerialNumber

    175 Analyze dependency resolution {#analyze-depedency-resolution}
    176 =============================
    177 
    178 When refactoring dependency handling or modifying variables used for dependency
    179 resolution, one may want to evaluate the impact on the existing applications.
    180 This describe some debug targets to dump variables used during dependency
    181 resolution.
    182 
    183 To analyze one board and application run the following commands in an
    184 application directory.
    185 
    186 Generate the variables dump with the normal dependency resolution to a
    187 `dependencies_info_board_name` file:
    188 
    189 ~~~~~~~~~~~~~~~~~~~
    190 BOARD=board_name make dependency-debug
    191 ~~~~~~~~~~~~~~~~~~~
    192 
    193 Or with the "quick" version used by murdock to know supported boards
    194 (this is an incomplete resolution, details in `makefiles/dependencies_debug.inc.mk`)
    195 to a `dependencies_info-boards-supported_board_name` file:
    196 
    197 ~~~~~~~~~~~~~~~~~~~
    198 BOARDS=board_name DEPENDENCY_DEBUG=1 make info-boards-supported
    199 ~~~~~~~~~~~~~~~~~~~
    200 
    201 For more configuration and usage details, see in the file defining the targets
    202 `makefiles/dependencies_debug.inc.mk`
    203 
    204 To do a repository wide analysis, you can use the script
    205 `dist/tools/buildsystem_sanity_check/save_all_dependencies_resolution_variables.sh`
    206 that will generate the output for all boards and applications.
    207 It currently take around 2 hours on an 8 cores machine with ssd.
