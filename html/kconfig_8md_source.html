    1 # Kconfig in RIOT {#kconfig-in-riot}
    2 
    3 [TOC]
    4 
    5 The objective of using Kconfig in RIOT is to configure software modules at
    6 compile-time. This means having a standard way of:
    7 - Exposing configurable parameters
    8 - Assigning application and user-specific configurations
    9 - Verifying these parameters
    10  - Check possible values
    11  - Check valid configuration considering inter-dependencies
    12 - Applying the selected configuration
    13 
    14 # Overview {#kconfig-overview}
    15 
    16 ## Exposure
    17 Modules in RIOT expose their configurable parameters via
    18 Kconfig files (for more information on Kconfig syntax check
    19 [the specification](https://www.kernel.org/doc/html/latest/kbuild/kconfig-language.html)).
    20 In these files documentation, restrictions, default values and dependencies can
    21 be expressed.
    22 
    23 Kconfig files are structured through the file system mirroring the current
    24 module distribution. In time, all modules will have Kconfig files to make
    25 themselves configurable through this system.
    26 
    27 ## Assignment
    28 The user can assign values to the exposed parameters, either by manually writing
    29 '.config' files or using an interface such as Menuconfig. Parameters with no
    30 assigned values will take the default ones. For a detailed distinction between
    31 Kconfig and '.config' files see [Appendix B](#kconfig-appendix-b).
    32 
    33 ## Verification and application
    34 Using '.config' and Kconfig files the build system takes care of doing the
    35 necessary checks on the values according to the parameter definition. After
    36 that, the `autoconf.h` header file is generated, it contains all the
    37 configurations in the form of (`CONFIG_` prefixed) macros.
    38

    40 # User's guide to configure with Kconfig {#kconfig-users-guide}
    41 
    42 ## Configure using menuconfig {#configure-using-menuconfig}
    43 In order to use the graphical interface menuconfig to configure the
    44 application, run `make menuconfig` in the application's folder. All available
    45 configurations (based on the used modules) for the particular platform will be
    46 presented. By default, the configuration of a module via Kconfig is not enabled.
    47 In order to activate the configuration via Kconfig the corresponding option
    48 should be selected. That will enable the configuration of all inner options, if
    49 available.
    50 
    51 Once the desired configuration is achieved save the configuration to the
    52 default proposed path and exit. The saved configuration will be applied when
    53 the code is compiled (`make all`).
    54 
    55 If the current configuration will be used in the future it can be saved in the
    56 application's folder as `user.config`, using the 'Save' option in menuconfig.
    57 This way it will be persistent after cleaning the application directory
    58 (`make clean`).
    59 
    60 ## Configure using '.config' files {#configure-using-files}
    61 The second way to configure the application is by directly writing '.config'
    62 files. Two files will be sources of configuration during the generation of the
    63 final header file: `app.config` and `user.config`, which should be placed
    64 inside the application's folder. `app.config` sets default configuration
    65 values for the particular application, the user can override them by setting
    66 them in `user.config`.
    67 
    68 Let's say that the `SOCK_UTIL_SCHEME_MAXLEN` symbol in `sock_util` module needs
    69 to be configured. The `user.config` file could look like:
    70 
    71 ```Makefile
    72 # activate configuration of sock_util using Kconfig
    73 CONFIG_KCONFIG_MODULE_SOCK_UTIL=y
    74 # change scheme part length
    75 CONFIG_SOCK_UTIL_SCHEME_MAXLEN=24
    76 ```
    77 
    78 In this case, there is no need for using menuconfig. It's enough just to call
    79 `make all` in the application folder, as this configuration will be read and
    80 applied. Note that if any dependency issue occurs, warnings will be generated
    81 (e.g. not enabling the configuration of a module via Kconfig).
    82 
    83 ## Application configuration with Kconfig {#app-config-kconfig}
    84 To expose application-specific configuration options a `Kconfig` file can
    85 be placed in the application's folder. For an example of this you can check
    86 the [tests/kconfig](https://github.com/RIOT-OS/RIOT/tree/master/tests/kconfig)
    87 application.
    88 
    89 ## A note on the usage of CFLAGS
    90 When a certain module is being configured via Kconfig the configuration macro
    91 will not longer be overridable by means of CFLAGS (e.g. set on the
    92 compilation command or on a Makefile). Consider this if you are getting a
    93 'redefined warning'.
    94 
    95 ## A note on the usage of the 'clean' command
    96 When using Kconfig as the configurator for RIOT, configuration symbols may be
    97 used in Makefiles through the build system. For this to work properly make
    98 sure that when cleaning an application you call `make clean && make all`,
    99 instead of `make clean all`.

    102 # Integration into the build system {#kconfig-integration-into-build-system}
    103 
    104 The integration of Kconfig into the build system is mainly done in
    105 `makefiles/kconfig.mk`.
    106 
    107 ## Steps during the build process {#kconfig-steps-build-process}
    108 
    109 ![Ouput of every step of the build process](kconfig_integration.svg)
    110 
    111 ### 0. Module dependency resolution
    112 Currently, the resolution of module dependencies is performed by the build
    113 system where all the used modules and packages end up listed in the `USEMODULE`
    114 make variables. In the next phases of integration we plan to resolve dependencies
    115 using Kconfig.
    116 
    117 
    118 #### Input
    119 - Makefiles.
    120 
    121 #### Output
    122 - `USEMODULE` and `USEPKG` variables.
    123 
    124 ### 1. Module listing
    125 The list of modules needed for the particular build is dumped into the
    126 `$ (GENERATED_DIR)/Kconfig.dep` file, where each module is translated into a
    127 Kconfig symbol as documented in [Appendix A](#kconfig-appendix-a).
    128 
    129 #### Input
    130 - `USEMODULE` and `USEPKG` variables
    131 
    132 #### Output
    133 - `$ (GENERATED_DIR)/Kconfig.dep` file
    134 
    135 ### 2. Merging all configuration sources {#kconfig-steps-merge-configs}
    136 In this step configuration values are taken from multiple sources and merged
    137 into a single `merged.config` configuration file. This file is temporary and is
    138 removed on clean. If the user needs to save a particular configuration
    139 set, a backup has to be saved (this can be done using the menuconfig interface)
    140 so it can be loaded later in this step.
    141 
    142 To accomplish merging of multiple input files, the `mergeconfig` script is
    143 used. Note that **the order matters**: existing configuration values are
    144 merged in the order expressed in the input section, where the last value
    145 assigned to a parameter has the highest priority. If no configuration files are
    146 available all default values will be applied.
    147 
    148 `merged.config` is the only configuration input for the `autoconf.h` in the
    149 [generation step](#kconfig-steps-header-gen).
    150 
    151 #### Input
    152 - Optional:
    153  - `$ (APPDIR)/app.config`: Application specific default configurations.
    154  - `$ (APPDIR)/user.config`: Configurations saved by user.
    155 
    156 #### Output
    157 - `$ (GENERATED_DIR)/merged.config` file.
    158 
    159 ### 3. Menuconfig execution (optional)
    160 Menuconfig is a graphical interface for software configuration. It is used for
    161 the configuration of the Linux kernel. This section explains the process
    162 that occurs when RIOT is being configured using the menuconfig interface.
    163 
    164 The main `Kconfig` file is used in this step to show the configurable
    165 parameters of the system. Kconfig will filter innaplicable parameters (i.e.
    166 parameters exposed by modules that are not being used) based on the file
    167 `$ (GENERATED_DIR)/Kconfig.dep` generated in step 1.
    168 
    169 During the transition phase, the user needs to enable Kconfig explicitly per
    170 module, by setting the corresponding option. If using `menuconfig` a checkbox
    171 with a submenu has to be selected, if using `.config` files a
    172 `CONFIG_KCONFIG_MODULE_` prefixed option has to be set to `y`. For more
    173 information see
    174 [Making configuration via Kconfig optional](#kconfig-configuration-optional).
    175 
    176 Note that if Kconfig is not used to configure a module, the corresponding
    177 header files default values will be used.
    178 
    179 `merged.config` is one of the inputs for menuconfig. This means that any
    180 configuration that the application defines in the `app.config` or a backup
    181 configuration from the user in `user.config` are taken into account on the
    182 first run (see [Appendix C](#kconfig-appendix-c)).
    183 
    184 In this step the user chooses configuration values (or selects the minimal
    185 configuration) and saves it to the `merged.config` file. Here the user can
    186 choose to save a backup configuration file for later at a different location
    187 (e.g. a `user.config` file in the application folder).
    188 
    189 #### Input
    190 - `/Kconfig` file.
    191 - Optional:
    192  - `$ (APPDIR)/app.config`
    193  - `$ (APPDIR)/user.config`
    194  - `$ (GENERATED_DIR)/merged.config`
    195 
    196 #### Output
    197 - Updated `$ (GENERATED_DIR)/merged.config` file.
    198 - `$ (GENERATED_DIR)/merged.config.old` backup file.
    199 
    200 ### 4. Generation of the autoconf.h header {#kconfig-steps-header-gen}
    201 With the addition of Kconfig a dependency has been added to the build
    202 process: the `$ (GENERATED_DIR)/autoconf.h` header file. This header file is
    203 the main output from the Kconfig configuration system. It holds all the macros
    204 that should be used to configure modules in RIOT:
    205 `CONFIG_<module>_<parameter>`.
    206 
    207 In order to generate the `autoconf.h` file the `genconfig` script is used.
    208 Inputs for this script are the main `Kconfig` file and `merged.config`
    209 configuration file, which holds the selected values for the exposed parameters.
    210 
    211 #### Input:
    212 - `$ (GENERATED_DIR)/merged.config` file.
    213 - Main `Kconfig` file exposing configuration of modules.
    214 
    215 #### Output:
    216 - `$ (GENERATED_DIR)/autoconf.h` configuration header file.
    217 - `$ (GENERATED_DIR)/out.config` file.
    218 
    219 
    220 ### Summary of files
    221 These files are defined in `kconfig.mk`.
    222 
    223 | File | Description |
    224 | ----------------- | ----------- |
    225 | `Kconfig` | Defines configuration options of modules. |
    226 | `Kconfig.dep` | Holds a list of the modules that are being compiled. |
    227 | `app.config` | Holds default application configuration values. |
    228 | `user.config` | Holds configuration values applied by the user. |
    229 | `merged.config` | Holds configuration from multiple sources. Used to generate header. |
    230 | `autoconf.h` | Header file containing the macros that applied the selected configuration. |
    231 | `out.config` | Configuration file containing all the symbols defined in `autoconf.h`. |
    232 
    233 ## Kconfig symbols in Makefiles
    234 As '.config' files have Makefile syntax they can be included when building,
    235 which allows to access the applied configuration from the build system and, in
    236 the future, to check for enabled modules.
    237 
    238 During migration this is also useful, as it gives the ability to check if a
    239 parameter is being configured via Kconfig or a default value via `CFLAGS` could
    240 be injected. For example:
    241 
    242 ```Makefile
    243 ifndef CONFIG_USB_VID
    244  CFLAGS += -DCONFIG_USB_VID=0x1209
    245 endif
    246 ```
    247 
    248 Symbols will have the same name as the configuration macros (thus will always
    249 have the `CONFIG_` prefix). As the configuration file is loaded in
    250 `Makefile.include` care should be taken when performing checks in the
    251 application's Makefile. The symbols will not be defined until after including
    252 `Makefile.include`.

    255 # Transition phase {#kconfig-transition-phase}
    256 ## Making configuration via Kconfig optional {#kconfig-configuration-optional}
    257 During transition to the usage of Kconfig as the main configurator for RIOT,
    258 the default behavior will be the traditional one: expose configuration options
    259 in header files and use CFLAGS as inputs. To allow optional configuration via
    260 Kconfig, a convention will be used when writing Kconfig files.
    261 
    262 Modules should be contained in their own `menuconfig` entries, this way the user
    263 can choose to enable the configuration via Kconfig for an specific module.
    264 These entries should define a dependency on the module they configure (see
    265 [Appendix A](#kconfig-appendix-a) to see how to check if a module is being
    266 used).
    267 
    268 The module configuration then can be enabled either via the menuconfig
    269 interface:
    270 
    271 ![menuconfig-example](kconfig_menuconfig.png)
    272 
    273 or by means of a '.config' file:
    274 
    275 ```Make
    276 CONFIG_KCONFIG_MODULE_GCOAP=y
    277 ```

    279 ## Modelling CPUs, boards and provided features {#kconfig-cpu-boards-features}
    280 During the current migration phase architectures, CPUs, boards and provided
    281 features are being modelled in Kconfig. The following is a guide on how to
    282 organize and name the symbols.
    283 
    284 ### Features
    285 Features must be modelled as hidden boolean symbols with the prefix `HAS_`. They
    286 must contain a `help` attribute clearly specifying what providing that feature
    287 means. The location of the symbol declaration depends on the type of feature.
    288 Features that are not platform-specific (e.g. `arch_32bit` or `cpp`) must be
    289 placed in `/kconfigs/Kconfig.features`. If a feature is specific to a certain
    290 CPU family or vendor, it should be placed in the correspondent Kconfig file
    291 (e.g. `esp_wifi_enterprise`). Features related to modules should be placed in
    292 the Kconfig file of that module.
    293 
    294 #### Example
    295 The feature `arduino` is placed in `/kconfigs/Kconfig.features` and modelled
    296 like:
    297 
    298 ```Kconfig
    299 config HAS_ARDUINO
    300  bool
    301  help
    302  Indicates that Arduino pins compatibility is supported.
    303 ```
    304 
    305 ### CPUs
    306 The proposed hierarchy for the classification of CPUs is as follows:
    307 
    308 ```
    309  +------------+
    310 More Specific | CPU_MODEL |
    311  + +------------+
    312  |
    313  |
    314  | +------------+
    315  | | CPU_FAM |
    316  | +------------+
    317  |
    318  |
    319  | +------------+
    320  | | CPU_CORE |
    321  | +------------+
    322  |
    323  |
    324  v +------------+
    325 Less Specific | CPU_ARCH |
    326  +------------+
    327 ```
    328 
    329 Where each hierarchy is defined as:
    330 - `CPU_MODEL`: The specific identifier of the used CPU, used for some CPU
    331  implementations to differentiate between different memory
    332  layouts.
    333 - `CPU_FAM`: An intermediate identifier between CPU and CPU_MODEL that
    334  represents a sub-group of a Manufacturers CPU's.
    335 - `CPU_CORE`: The specific identifier of the core present in the CPU.
    336 - `CPU_ARCH`: The specific identifier of the architecture of the core defined
    337  in `CPU_CORE`.
    338 
    339 In order to model the hierarchies, a hidden boolean symbol must be declared for
    340 each. The name of the symbol must begin with the correspondent prefix and must
    341 be followed by the specific value. For instance, the 'samd21' family symbol is
    342 named `CPU_FAM_SAMD21`.
    343 
    344 In addition, a default value to the correspondent common symbol must be defined.
    345 The default value must be guarded by the boolean symbol correspondent to the
    346 hierarchy.
    347 
    348 Features may be provided by any hierarchy symbol. Usually symbols are selected
    349 from more specific to less specific. This means that a `CPU_MODEL_<model>`
    350 symbol usually would select the correspondent `CPU_FAM_<family>` symbol,
    351 which would in turn select the `CPU_CORE_<core>`. This may change in some cases
    352 where `CPU_COMMON_` symbols are defined to avoid repetition.
    353 
    354 In addition to the symbols of the hierarchy described above, a default value
    355 to the `CPU` symbol should be assigned, which will match the value of the `CPU`
    356 Makefile variable in the build system.
    357 
    358 The declaration of the symbols should be placed in a `Kconfig` file in the
    359 folder that corresponds to the hierarchy. When the symbols are scattered into
    360 multiple files, it is responsibility of file containing the most specific
    361 symbols to `source` the less specific. Keep in mind that only the file located
    362 in `/cpu/<CPU>/Kconfig` will be included by the root `/Kconfig` file.
    363 
    364 #### Example
    365 
    366 ```Kconfig
    367 # This is the most specific symbol (selected by the board)
    368 # The CPU model selects the family it belongs to
    369 config CPU_MODEL_SAMR21G18A
    370  bool
    371  select CPU_FAM_SAMD21
    372 
    373 # In this case the family selects a common 'sam0' symbol (which provides some
    374 # features), and the core it has (cortex-m0+)
    375 config CPU_FAM_SAMD21
    376  bool
    377  select CPU_COMMON_SAM0
    378  select CPU_CORE_CORTEX_M0PLUS
    379  select HAS_CPU_SAMD21
    380  select HAS_PUF_SRAM
    381 
    382 # The value of the common value depends on the selected model
    383 config CPU_MODEL
    384  default "samd21e18a" if CPU_MODEL_SAMD21E18A
    385  default "samd21g18a" if CPU_MODEL_SAMD21G18A
    386  default "samd21j18a" if CPU_MODEL_SAMD21J18A
    387  default "samr21e18a" if CPU_MODEL_SAMR21E18A
    388  default "samr21g18a" if CPU_MODEL_SAMR21G18A
    389 
    390 config CPU_FAM
    391  default "samd21" if CPU_FAM_SAMD21
    392 
    393 ```
    394 
    395 ### Boards
    396 Boards must be modelled as hidden boolean symbols with the prefix `BOARD_` which
    397 default to `y` and are placed in `/boards/<BOARD>/Kconfig`. This file will be
    398 `source`d from the main `/Kconfig` file. The board symbol must select the
    399 `CPU_MODEL_<model>` symbol that corresponds to the CPU model present on the
    400 board. The board symbol must also select the symbols that correspond to the
    401 features it provides.
    402 
    403 In the same `Kconfig` file a default value must be assigned to the
    404 common `BOARD` symbol. It must be guarded by the board's symbol, so it only
    405 applies in that case.
    406 
    407 There are cases when grouping common code for multiple boards helps to avoid
    408 unnecessary repetition. In the case features are provided in a common board
    409 folder (e.g. `/boards/common/arduino-atmega`) a symbol should be declared to
    410 model this in Kconfig. Symbols for common boards must have the `BOARD_COMMON_`
    411 prefix, and must select the common provided features.
    412 
    413 #### Example
    414 The samr21-xpro has a `samr21g18a` CPU and provides multiple features. Its
    415 symbol is modelled as following:
    416 
    417 ```Kconfig
    418 # /boards/samr21-xpro/Kconfig
    419 
    420 config BOARD
    421  default "samr21-xpro" if BOARD_SAMR21_XPRO
    422 
    423 config BOARD_SAMR21_XPRO
    424  bool
    425  default y
    426  select CPU_MODEL_SAMR21G18A
    427  select HAS_PERIPH_ADC
    428  select HAS_PERIPH_I2C
    429  select HAS_PERIPH_PWM
    430  select HAS_PERIPH_RTC
    431  select HAS_PERIPH_RTT
    432  select HAS_PERIPH_SPI
    433  select HAS_PERIPH_TIMER
    434  select HAS_PERIPH_UART
    435  select HAS_PERIPH_USBDEV
    436  select HAS_RIOTBOOT
    437 ```
    438 
    439 ## Summary of reserved Kconfig prefixes
    440 The following symbol prefixes have been assigned particular semantics and are
    441 reserved for the cases described bellow:
    442 
    443 <!-- Keep the table in alphabetical order -->
    444 | Prefix | Description |
    445 | :----- | :---------- |
    446 | `BOARD_` | Models a board |
    447 | `BOARD_COMMON_` | Used for common symbols used by multiple boards |
    448 | `CPU_ARCH_` | Models a CPU architecture |
    449 | `CPU_COMMON_` | Used for common symbols used by multiple CPUs |
    450 | `CPU_CORE_` | Models a CPU core |
    451 | `CPU_FAM_` | Models a family of CPUs |
    452 | `CPU_MODEL_` | Models a particular model of CPU |
    453 | `HAS_` | Models a [feature](build-system-basics.html#features) |
    454 | `KCONFIG_MODULE_` | Used during transition to enable configuration of a module via Kconfig |
    455 | `KCONFIG_PKG_` | Used during transition to enable configuration of a package via Kconfig |
    456 | `MODULE_` | Models a [RIOT module](creating-modules.html#creating-modules) |
    457 | `PKG_` | Models an [external package](group__pkg.html) |

    460 # Appendixes {#kconfig-appendixes}
    461 
    462 ## Appendix A: Check if a module or package is used {#kconfig-appendix-a}
    463 In order to show only the relevant configuration parameters to the user with
    464 respect to a given application and board selection, Kconfig needs knowledge
    465 about all modules and packages to be used for a compilation. Currently
    466 dependency handling among modules is performed by the build system (via
    467 `Makefile.dep` files). The interface defined to declared the used modules and
    468 packages is the `$ (GENERATED_DIR)/Kconfig.dep` file.
    469 
    470 `Kconfig.dep` is a Kconfig file that will define symbols of the form:
    471 ```Kconfig
    472 config MODULE_SOCK_UTIL
    473  bool
    474  default y
    475 ```
    476 There will be a symbol for every used module (i.e. every module in
    477 `USEMODULE` make variable) and package. The names in the symbols will be
    478 uppercase and separated by `_`. Based on these symbols configurability is
    479 decided. Modules and packages symbols will have `MODULE_` and `PKG_` prefixes
    480 respectively.
    481 
    482 The following is an example of how to use these symbols in Kconfig files to
    483 enable/disable a configuration menu:
    484 
    485 ```Kconfig
    486 menuconfig KCONFIG_MODULE_SOCK_UTIL
    487  bool "Configure Sock Utilities"
    488  depends on MODULE_SOCK_UTIL
    489  help
    490  "Configure Sock Utilities using Kconfig."
    491 ```
    492 
    493 Then, every configuration option for the previous module would be modeled like:
    494 
    495 ```Kconfig
    496 if KCONFIG_MODULE_SOCK_UTIL
    497 
    498 config SOCK_UTIL_SCHEME_MAXLEN
    499  int "Maximum length of the scheme part for sock_urlsplit"
    500  default 16
    501 
    502 endif # KCONFIG_MODULE_SOCK_UTIL
    503 ```

    505 ## Appendix B: Difference between 'Kconfig' and '.config' files {#kconfig-appendix-b}
    506 Kconfig files describe a configuration database, which is a collection of
    507 configuration options organized in a tree structure. Configuration options may
    508 have dependencies (among other attributes), which are used to determine their
    509 visibility.
    510 
    511 Kconfig files are written in
    512 [Kconfig language](https://www.kernel.org/doc/Documentation/kbuild/kconfig-language.txt)
    513 defined in the Linux kernel. Configuration options have attributes such as
    514 types, prompts and default values.
    515 
    516 #### Kconfig file
    517 
    518 ```Kconfig
    519 menu "Buffer Sizes"
    520 
    521 config GCOAP_PDU_BUF_SIZE
    522  int "Request or response buffer size"
    523  default 128
    524 
    525 endmenu
    526 ```
    527 
    528 On the other hand configuration files contain assignment of values to
    529 configuration options and use Makefile syntax. They can also be used to save a
    530 set of configuration values as backup.
    531 
    532 #### '.config' file
    533 
    534 ```makefile
    535 # enable Kconfig configuration for gcoap
    536 CONFIG_KCONFIG_MODULE_GCOAP=y
    537 # set the value
    538 CONFIG_GCOAP_PDU_BUF_SIZE=12345
    539 ```
    540 
    541 In other words: Kconfig files describe configuration options and '.config' files
    542 assign their values.
    543 
    544 ## Appendix C: Pitfall when using different configuration interfaces {#kconfig-appendix-c}
    545 In the current configuration flow the user can choose to configure RIOT using
    546 the menuconfig graphical interface or writing '.config' files by hand.
    547 
    548 As explained in the
    549 ['Configuration sources merging step'](#kconfig-steps-merge-configs)
    550 of the configuration process, configuration from multiple sources are loaded to
    551 create a single `merged.config` file, and the order of merging matters: last
    552 file has priority.
    553 
    554 While editing values directly via '.config' files `merged.config` will be
    555 re-built. Once the user decides to edit `merged.config` directly using
    556 menuconfig, the file will not be re-built anymore, and any changes by manually
    557 editing the source files will have no effect. To go back to manual edition
    558 a `make clean` has to be issued in the application directory.
    559 
    560 ## Appendix D: A few key aspects while exposing a macro to Kconfig {#kconfig-appendix-d}
    561 A macro that holds a 0 or 1 is modelled in Kconfig as a `bool` symbol. References to this macro
    562 can then make use of IS_ACTIVE macro from kernel_defines.h with C conditionals
    563 for conditional compilation.
    564 [FXOS8700 driver exposure to Kconfig](https://github.com/RIOT-OS/RIOT/pull/13914)
    565 can be considered as an example. If the macro is defined as `TRUE` by default,
    566 a new symbol gets introduced to invert the semantics. The recommended
    567 practice is to add a new symbol and expose it to Kconfig while the old one is
    568 tagged to be deprecated. The process is documented in this
    569 [commit](https://github.com/RIOT-OS/RIOT/pull/13129/commits/c7b6dc587cf20f3177abe0417a408b6ab90d0ff8)
    570 
    571 There may be cases where a macro is expected to hold only specific values, e.g.
    572 'GNRC_IPV6_MSG_QUEUE_SIZE' expressed as the power of two. These may be modelled
    573 in such a way that a new macro is introduced to hold the restricted figures
    574 while operators are added to arrive at the desired value. The process is
    575 documented in this [pull request.](https://github.com/RIOT-OS/RIOT/pull/14086)
