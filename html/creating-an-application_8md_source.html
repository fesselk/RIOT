    1 Creating an application {#creating-an-application}
    2 =======================
    3 
    4 [TOC]
    5 
    6 To create your own application you need to create a directory containing one or
    7 multiple C file(s) with your source code and a Makefile. A template Makefile is
    8 available in the `dist` folder of the
    9 [RIOT repository](https://github.com/RIOT-OS/RIOT).
    10 
    11 The main function {#the-main-function}
    12 =================
    13 After the board is initialized, RIOT starts two threads: the idle thread and
    14 the main thread. The idle thread has the lowest priority and will run whenever
    15 no other thread is ready to run. It will automatically use the lowest possible
    16 power mode for the device. The main thread - configured with a default priority
    17 that is right in the middle between the lowest and the highest available
    18 priorities - is the first thread that runs and calls the `main()` function.
    19 This function needs to be defined in the source code of the application
    20 (typically located in `main.c`).
    21 
    22 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.c}
    23 #include <stdio.h>
    24 
    25 int main(void)
    26 {
    27  puts("Hello World!");
    28  return 0;
    29 }
    30 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

    32 The application's Makefile {#the-applications-makefile}
    33 ==========================
    34 The minimal Makefile {#the-minimal-makefile}
    35 --------------------
    36 At minimum the Makefile of an application (see @ref getting-started) needs to
    37 define the following macros:
    38  * `APPLICATION`: should contain the name of your application
    39  * `RIOTBASE`: specifies the path to your copy of the RIOT repository (note
    40  that you may want to use `$(CURDIR)` here, to give a relative path)
    41 
    42 The `BOARD` macro is also required and recommended to be set to `native` by
    43 default, but is recommended to be overridable with the `?=` operator.
    44 Additionally, it is required to include the `Makefile.include` from the
    45 `RIOTBASE`.
    46 
    47 ~~~~~~~~~~~~~~~~~~~~~~~~~~~ {.mk}
    48 # Set the name of your application:
    49 APPLICATION = foobar
    50 
    51 # If no BOARD is found in the environment, use this default:
    52 BOARD ?= native
    53 
    54 # This has to be the absolute path to the RIOT base directory:
    55 RIOTBASE ?= $(CURDIR)/../../RIOT
    56 
    57 include $(RIOTBASE)/Makefile.include
    58 ~~~~~~~~~~~~~~~~~~~~~~~~~~~

    60 How to handle unsupported boards? {#handle-unsupported-boards}
    61 ---------------------------------
    62 Sometimes it is necessary to exclude boards because they don't provide a
    63 required functionality or don't have sufficient memory. RIOT's build system
    64 looks for the macros `BOARD_BLACKLIST`, `BOARD_WHITELIST`, and
    65 `BOARD_INSUFFICIENT_MEMORY`. Any board name that is not included in
    66 `BOARD_WHITELIST` will issue a message that one has to expect errors if they
    67 build the application for the board referred by that name. The list can also be
    68 used by a CI system to not build the application for this board at all. A board
    69 that is included in `BOARD_BLACKLIST` will show the same behavior. The build
    70 system evaluates `BOARD_WHITELIST` first and then `BOARD_BLACKLIST`. The
    71 `BOARD_INSUFFICIENT_MEMORY` macro is similar to `BOARD_BLACKLIST` but will
    72 build in any case. A CI system can use the information provided by the
    73 `BOARD_INSUFFICIENT_MEMORY` macro to skip the linking step in the build
    74 process, since some linkers will issue an error if the code won't fit the
    75 target board's flash memory or RAM.

    77 Including modules {#including-modules}
    78 -----------------
    79 By default a RIOT application comprises only of the applications' code itself,
    80 the kernel, and platform specific code. In order to use additional modules,
    81 such as a particular [device driver](@ref drivers) or [a system library](@ref sys)
    82 (including [networking capabilities](@ref net)), you have to append the modules'
    83 names to the USEMODULE variable. For example, to build an application using the
    84 SHT11 temperature sensor and UDP/IPv6 functionalities of the GNRC network stack,
    85 your Makefile needs to contain these lines:
    86 
    87 ~~~~~~~~~~~~~~~~~~~~~~~~~~~ {.mk}
    88 USEMODULE += sht11
    89 USEMODULE += gnrc_ipv6_default
    90 USEMODULE += gnrc_udp
    91 ~~~~~~~~~~~~~~~~~~~~~~~~~~~
    92 
    93 Modules typically pull in all required dependencies.

    95 # Helper tools
    96 
    97 To help you start writing an application within RIOT, the build system provides
    98 the `generate-example` `generate-test` make targets. These targets are wrappers
    99 around the [riotgen](https://pypi.org/project/riotgen/) command line tool and
    100 are helpful when starting to implement an application: all required files are
    101 generated with copyright headers, doxygen groups, etc, so you can concentrate
    102 on the module implementation.
    103 For applications, the `Makefile` is generated with the dependencies (modules,
    104 packages, required features) included.
    105 
    106 **Usage:**
    107 
    108 To generate an example application, e.g in the `examples` directory, from the
    109 RIOT base directory, run:
    110 ```
    111 make generate-example
    112 ```
    113 
    114 To generate a test application, e.g in the `tests` directory, from the
    115 RIOT base directory, run:
    116 ```
    117 make generate-test
    118 ```
    119 
    120 Then answer a few questions about the application:
    121 - Application name: enter a name for your application. It will be used as both
    122  the name of the application directory under `examples` or `tests` and by
    123  the build system module (set in the `APPLICATION` variable).
    124 - Application brief description: Describe in one line what is this application
    125  about.
    126 - Target board: select the default target board. The value is set to `native`
    127  by default.
    128 - Modules: give the list of dependency modules, separated by commas. For
    129  example: ztimer,fmt
    130 - Packages: give the list of dependency packages, separated by commas.
    131 - Features required: give the list of CPU features (`periph_*`, etc) required
    132  by the application, all separated by commas.
    133 
    134 Other global information (author name, email, organization) should be retrieved
    135 automatically from your git configuration.
    136 
    137 Once completed, the application files are either located in
    138 `examples/<application name>` or `tests/<application name>` depending on the
    139 target used.
    140 
    141 **Testrunner:** when using the `make generate-test`, you can also automatically
    142 add a testrunner Python script. Just answer 'y' when prompted.
