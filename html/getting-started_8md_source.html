    6 Downloading RIOT code {#downloading-riot-code}
    7 =====================
    8 You can obtain the latest RIOT code from
    9 our [Github](https://github.com/RIOT-OS/) repository either by
    10 [downloading the latest tarball](https://github.com/RIOT-OS/RIOT/releases) or
    11 by cloning the [git repository](https://github.com/RIOT-OS/RIOT).
    12 
    13 In order to clone the RIOT repository, you need the
    14 [Git revision control system](http://git-scm.com/) and run the following
    15 command:
    16 
    17 ~~~~~~~~ {.sh}
    18 git clone git://github.com/RIOT-OS/RIOT.git
    19 ~~~~~~~~

    24 Setting up a toolchain {#setting-up-a-toolchain}
    25 ----------------------
    26 Depending on the hardware you want to use, you need to first install a
    27 corresponding toolchain. The Wiki on RIOT's Github page contains a lot of
    28 information that can help you with your platform:
    29 
    30 * [ARM-based platforms](https://github.com/RIOT-OS/RIOT/wiki/Family:-ARM)
    31 * [TI MSP430](https://github.com/RIOT-OS/RIOT/wiki/Family:-MSP430)
    32 * [Atmel ATmega](https://github.com/RIOT-OS/RIOT/wiki/Family%3A-ATmega)
    33 * [native](https://github.com/RIOT-OS/RIOT/wiki/Family:-native)

    35 The build system {#the-build-system}
    36 ----------------
    37 RIOT uses [GNU make](https://www.gnu.org/software/make/) as build system. The
    38 simplest way to compile and link an application with RIOT, is to set up a
    39 Makefile providing at least the following variables:
    40 
    41  * `APPLICATION`: should contain the (unique) name of your application
    42  * `BOARD`: specifies the platform the application should be built for by
    43  default
    44  * `RIOTBASE`: specifies the path to your copy of the RIOT repository (note,
    45  that you may want to use `$(CURDIR)` here, to give a relative path)
    46 
    47 Additionally it has to include the `Makefile.include`, located in RIOT's root
    48 directory:
    49 
    50 ~~~~~~~~ {.mk}
    51 # a minimal application Makefile
    52 APPLICATION = mini-makefile
    53 BOARD ?= native
    54 RIOTBASE ?= $(CURDIR)/../RIOT
    55 
    56 include $(RIOTBASE)/Makefile.include
    57 ~~~~~~~~
    58 
    59 You can use Make's `?=` operator in order to allow overwriting
    60 variables from the command line. For example, you can easily specify the target
    61 platform, using the sample Makefile, by invoking make like this:
    62 
    63 ~~~~~~~~ {.sh}
    64 make BOARD=iotlab-m3
    65 ~~~~~~~~
    66 
    67 Besides typical targets like `clean`, `all`, or `doc`, RIOT provides the
    68 special targets `flash` and `term` to invoke the configured flashing and
    69 terminal tools for the specified platform. These targets use the variable
    70 `PORT` for the serial communication to the device. Neither this variable nor
    71 the targets `flash` and `term` are mandatory for the native port.
    72 
    73 For the native port, `PORT` has a special meaning: it is used to identify the
    74 tap interface if the `netdev_tap` module is used. The target `debug` can be
    75 used to invoke a debugger on some platforms. For the native port the additional
    76 targets such as `all-valgrind` and `valgrind` exist. Refer to
    77 `cpu/native/README.md` for additional information
    78 
    79 Some RIOT directories contain special Makefiles like `Makefile.base`,
    80 `Makefile.include` or `Makefile.dep`. The first one can be included into other
    81 Makefiles to define some standard targets. The files called `Makefile.include`
    82 are used in `boards` and `cpu` to append target specific information to
    83 variables like `INCLUDES`, setting the include paths. `Makefile.dep` serves to
    84 define dependencies.
    85 
    86 Unless specified otherwise, make will create an elf-file as well as an Intel
    87 hex file in the `bin` folder of your application directory.
    88 
    89 Learn more about the build system in the
    90 [Wiki](https://github.com/RIOT-OS/RIOT/wiki/The-Make-Build-System)

    92 Building and executing an example {#building-and-executing-an-example}
    93 ---------------------------------
    94 RIOT provides a number of examples in the `examples/` directory. Every example
    95 has a README that documents its usage and its purpose. You can build them by
    96 typing
    97 
    98 ~~~~~~~~ {.sh}
    99 make BOARD=samr21-xpro
    100 ~~~~~~~~
    101 
    102 or
    103 
    104 ~~~~~~~~ {.sh}
    105 make all BOARD=samr21-xpro
    106 ~~~~~~~~
    107 
    108 into your shell.
    109 
    110 To flash the application to a board just type
    111 
    112 ~~~~~~~~ {.sh}
    113 make flash BOARD=samr21-xpro
    114 ~~~~~~~~
    115 
    116 You can then access the board via the serial interface:
    117 
    118 ~~~~~~~~ {.sh}
    119 make term BOARD=samr21-xpro
    120 ~~~~~~~~
    121 
    122 If you are using multiple boards you can use the `PORT` macro to specify the
    123 serial interface:
    124 
    125 ~~~~~~~~ {.sh}
    126 make term BOARD=samr21-xpro PORT=/dev/ttyACM1
    127 ~~~~~~~~
    128 
    129 Note that the `PORT` macro has a slightly different semantic in `native`. Here
    130 it is used to provide the name of the TAP interface you want to use for the
    131 virtualized networking capabilities of RIOT.
    132 
    133 We use `pyterm` as the default terminal application. It is shipped with RIOT in
    134 the `dist/tools/pyterm/` directory. If you choose to use another terminal
    135 program you can set `TERMPROG` (and if need be the `TERMFLAGS`) macros:
    136 
    137 ~~~~~~~~ {.sh}
    138 make -C examples/gnrc_networking/ term \
    139  BOARD=samr21-xpro \
    140  TERMPROG=gtkterm \
    141  TERMFLAGS="-s 115200 -p /dev/ttyACM0 -e"
    142 ~~~~~~~~

    144 Configuring an application {#configuring-an-application}
    145 --------------------------
    146 Many modules in RIOT offer configuration options that will be considered during
    147 compile-time.They are modeled as macros that can be overridden by the user.
    148 Currently there are two ways of doing this: using `CFLAGS` or via
    149 @ref kconfig-in-riot "Kconfig" (the last one is currently only possible for a
    150 subset of modules).
    151 
    152 For instructions on how to configure via `CFLAGS` check the
    153 @ref config "identified compile-time configurations". To learn how to use
    154 Kconfig in RIOT, please refer to the @ref kconfig-users-guide.
