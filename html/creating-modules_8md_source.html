    1 Creating modules {#creating-modules}
    2 ================
    3 Modules in RIOT are well-defined units of code that provide a set of features
    4 to your application. This includes also drivers and to a certain extent ports
    5 for CPUs and boards (with some exceptions, see <!-- @ref porting-guide --> the
    6 porting guide for further information).
    7 
    8 The general structure {#the-general-structure}
    9 =====================
    10 Like @ref creating-an-application "applications", modules are directories
    11 containing source files and a Makefile. Additionally their API can be defined
    12 in one or more header files, residing in the include path of their
    13 super-module.
    14 
    15 E.g. the @ref sys_shell module is implemented in `sys/shell` and defines its
    16 API in `sys/include/shell.h` and the @ref drivers_isl29020 driver is
    17 implemented in `drivers/isl29020` and defines its API in
    18 `drivers/include/isl29020.h`.
    19 
    20 A module's Makefile just needs to include `Makefile.base` in the RIOT
    21 repository:
    22 
    23 ~~~~~~~~~~~~~~~~~~~ {.mk}
    24 include $(RIOTBASE)/Makefile.base
    25 ~~~~~~~~~~~~~~~~~~~
    26 
    27 If your module's name differs from the name of the directory it resides in you
    28 need to set the `MODULE` macro in addition.
    29 
    30 When compiled a module always provides a `MODULE_<MODULENAME>` macro to the
    31 system. This way, other modules can check if the module is available in the
    32 current configuration or not.
    33 
    34 Modules can be used by adding their name to the `USEMODULE` macro of your
    35 application's Makefile.
    36 
    37 ### Pitfalls ###
    38 
    39 The `MODULE` name should be unique or build breaks as modules overwrite the
    40 same output file. This might for example lead to `undefined reference to` errors
    41 in the linker which can be hard to track down.
    42 
    43 This problem happened in the past for:
    44 
    45  * Packages root directory (libfixmath/u8g2)
    46  * boards/cpu/periph and their common boards/cpu/periph
    47 
    48 Note: even if all boards and cpus implement the `board` and `cpu` modules, only
    49  one is used in an application so there is no conflict.
    50 
    51 
    52 Module dependencies
    53 ===================
    54 Your module may depend on other modules to minimize code duplication. These
    55 dependencies are defined in `Makefile.dep` with the following syntax:
    56 
    57 ~~~~~~~~~~~~~~~~~~~ {.mk}
    58 ifneq (,$(filter your_module,$(USEMODULE))) # if module in USEMODULE
    59  USEMODULE += dep1 # add dependencies to USEMODULE
    60  USEMODULE += dep2
    61 endif
    62 ~~~~~~~~~~~~~~~~~~~
    63 
    64 Note, that `Makefile.dep` is processed only once so you have to take care to
    65 add the dependency block for your module *before* your dependencies pull in
    66 their dependencies.
    67 
    68 Modules outside of RIOTBASE {#modules-outside-of-riotbase}
    69 ===========================
    70 Modules can be defined outside `RIOTBASE`. In addition to add it to `USEMODULE`
    71 the user needs to add the module path to `EXTERNAL_MODULE_DIRS`.
    72 
    73 The external module can optionally define the following files:
    74 * `Makefile.include` file to set global build configuration like `CFLAGS` or add
    75  API headers include paths to the `USEMODULE_INCLUDES` variable.
    76 * `Makefile.dep` file to set module dependencies
    77 
    78 An example can be found in
    79 [`tests/external_module_dirs`](https://github.com/RIOT-OS/RIOT/tree/master/tests/external_module_dirs)
    80 
    81 Pseudomodules {#pseudomodules}
    82 =============
    83 Pseudomodules are modules that are not static libraries, i.e. do not generate a
    84 `<module name>.a` file.
    85 
    86 To create a pseudomodule just add its name to `makefiles/pseudomodules.inc.mk`
    87 with `PSEUDOMODULES += <modulename>` in alphabetical order.
    88 
    89 A Pseudomodule may or may not have a source file associated with it. To make the
    90 distinction between them we will refer to those that don't as true-Pseudomodules.
    91 
    92 The main use case for true-Pseudomodules is to provide base information for
    93 dependencies to other modules or information to the code base via the
    94 `MODULE_<MODULENAME>` macro.
    95 
    96 Pseudomodules with source code exist under a "real" `MODULE` since they will
    97 generate a `<pseudomodule_name>.o` file grouped under that `MODULE`s
    98 `<module_name>.a` file.
    99 
    100 These modules appear in RIOT under two forms:
    101 
    102 1. Conditionally included source files:
    103 
    104  ```
    105  foo/
    106  |----foo_bar.c
    107  |----foo.c
    108  |----Makefile
    109  ```
    110 
    111 In `foo/Makefile` you add the source file to the `SRC` variable, conditioned on
    112 the Pseudomodule inclusion
    113 
    114  ```mk
    115  ifneq (,$(filter foo_bar,$(USEMODULE)))
    116  SRC += foo_bar.c
    117  endif
    118  ```
    119 
    120 See `sys/net/ble/skald` for an example in code.
    121 
    122 2. Using the `SUBMODULES` mechanism:
    123 
    124  ```
    125  foo/
    126  |----spam.c
    127  |----ham.c
    128  |----eggs.c
    129  |----Makefile
    130  ```
    131 
    132  ```mk
    133  # make all code end up in "foo_bar.a", this can be any name
    134  MODULE := foo_bar
    135 
    136  # ensure that "foo_ham" or "bar_foo_ham" builds "foo_ham.c".
    137  BASE_MODULE := foo
    138 
    139  # list of source files that are not SUBMODULES
    140  SRC := spam.c
    141 
    142  # enable submodules by setting SUBMODULES = 1
    143  SUBMODULES = 1
    144  ```
    145 
    146 When using `SUBMODULES` in a `MODULE` all `SRC` file excluded from `foo/Makefile`
    147 will be considered `SUBMODULES`. In the example above `ham.c` and `eggs.c`.
    148 These source files will be conditionally included depending if the modules have
    149 been added, i.e. `USEMODULE += foo_ham foo_eggs` (it's the same as case 1 but
    150 handled automatically in `Makefile.base`).
    151 
    152 The `SUBMODULES` mechanism is more flexible since `BASE_MODULE` allows matching
    153 the only parts of compounded module names and only match against part of that name.
    154 
    155 See `sys/ztimer/Makefile` for an example in code.
    156 
    157 `SUBMODULES` can also be true-pseudomodules.

    159 # Helper tools
    160 
    161 To help you start writing a module, the RIOT build system provides the
    162 `generate-module` make target. It is a wrapper around the
    163 [riotgen](https://pypi.org/project/riotgen/) command line tool that is helpful
    164 when starting to implement a module: all required files are generated with
    165 copyright headers, doxygen groups, etc, so you can concentrate on the module
    166 implementation.
    167 The module source files are created in the `sys` directory.
    168 
    169 **Usage:**
    170 
    171 From the RIOT base directory, run:
    172 ```
    173 make generate-module
    174 ```
    175 Then answer a few questions about the driver:
    176 - Module name: enter a name for your module. It will be used as both the name
    177  of the module directory under sys, where the source files are created, and
    178  the build system module (used with `USEMODULE`).
    179 - Module doxygen name: Enter the name of module, as displayed in the
    180  Doxygen documentation.
    181 - Brief doxygen description: Describe in one line what is this module about.
    182 
    183 Other global information (author name, email, organization) should be retrieved
    184 automatically from your git configuration.
    185 
    186 Once completed, the module files are located in
    187 `sys/<module name>/<module name>.c` and `sys/include/<module name>.h`.
